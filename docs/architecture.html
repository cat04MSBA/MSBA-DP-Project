🥇 Curated / Gold +
usage-driven materialized views
What "gold" means here
·
Materialized views derived from popular query patterns. Every researcher query in the Streamlit app is logged. When the same filter combination shows up frequently — same metric set, same country group, same year range — we promote it to a pre-aggregated materialized view that answers in milliseconds instead of re-scanning the 3M-row silver table.
How it works
①
Log every query the Streamlit app fires · stored in ops.query_log · captures filter signature, row count returned, response time
②
Periodic review of the log surfaces query patterns that hit a frequency threshold (e.g. same filter signature run > 20 times in a week)
③
Promote to materialized view · a new view is created in the curated schema with the result pre-computed, refreshed by the Coverage task on every successful transformation run
④
App routes transparently · the next time that query pattern arrives, the app reads from the curated view instead of silver — sub-100ms response
Why usage-driven
·
Materializing every possible aggregation would explode storage and refresh cost. Letting actual researcher behaviour drive what gets materialized means we only pay to pre-compute the slices that matter — and we automatically discover patterns we wouldn't have predicted in advance.
